# RFC-002: relate lint — Unified Linting via cypher-data and gram-data

## Summary

`relate lint` is a unified linting command for `.cypher` and `.gram` files.
Rather than implementing lint rules directly, `relate` wraps the linting engines
developed in `cypher-data` and `gram-data`. This RFC specifies the `relate lint`
interface, the dispatch logic by file type, and the API surface each upstream lib
must expose so that `relate` can call them programmatically.

## Motivation

`cypher-data` and `gram-data` are the canonical implementations of Cypher and Gram
linting respectively. Their rule sets (`.scm` query files), diagnostics, and
extensibility mechanisms will evolve independently of `relate`. `relate` should not
duplicate or fork this logic — it should be a consumer.

`relate lint` adds value at the orchestration layer:
- A single entry point for both file types in a mixed project
- Integration with the broader `relate` workflow (e.g., lint before write)
- Consistent output format regardless of which engine ran
- A foundation for CI integration across Cypher and Gram files

## Design

### CLI Interface

The `relate lint` interface (already established in `src/cli.rs`) is:

```
relate lint [OPTIONS] [FILES]...

Arguments:
  [FILES]...   Files or directories to lint (.cypher or .gram).
               Reads from stdin if omitted.

Options:
  -e, --expr <EXPR>   Lint an inline expression instead of a file
      --lang <LANG>   Language for --expr or stdin [default: cypher]
                      [possible values: cypher, gram]
      --json          Output diagnostics as JSON
      --strict        Treat warnings as errors
  -h, --help          Print help
```

The `--lang` flag is new relative to RFC-001's stub. It disambiguates the input
language when linting an inline expression or stdin, where a file extension is not
available. Default is `cypher` since inline Cypher snippets are the common case.

### Dispatch Logic

`relate lint` dispatches by file extension:

| Extension | Engine | Upstream command |
|-----------|--------|-----------------|
| `.cypher` | `cypher-data` lint engine | `cypher lint` |
| `.gram`   | `gram-data` lint engine | `gram lint` (renamed from `gram check` in 0.3.9; resolves [tree-sitter-gram#10](https://github.com/gram-data/tree-sitter-gram/issues/10)) |
| other     | Error: unsupported file type | — |

When a directory is given, `relate lint` walks it recursively and dispatches each
`.cypher` and `.gram` file to the appropriate engine. Other files are silently
skipped (not an error).

For `--expr` and stdin, the `--lang` flag selects the engine.

### Required API: cypher-data lib

`cypher-data` 0.2.2 exposes the following public surface used by `relate`:

```rust
// Actual public API in cypher-data 0.2.2

pub mod lint {
    pub struct LintOptions {
        pub strict: bool,
    }

    // Diagnostic is gram_diagnostics::Diagnostic, re-exported via cypher-data
    pub fn lint_source(source: &str, options: &LintOptions) -> Vec<gram_diagnostics::Diagnostic>;
    pub fn lint_file(path: &std::path::Path, options: &LintOptions)
        -> anyhow::Result<Vec<gram_diagnostics::Diagnostic>>;
}
```

Note: `cypher-data` re-exports `gram_diagnostics::Diagnostic`; `relate` uses that type
directly without conversion.

### Required API: gram-data lib

`gram-data` 0.3.9 exposes the following public surface used by `relate`:

```rust
// Actual public API in gram-data 0.3.9

pub mod lint {
    pub struct LintOptions {
        pub strict: bool,
    }

    // Diagnostic is gram_diagnostics::Diagnostic, re-exported via gram-data
    pub fn lint_source(source: &str, options: &LintOptions) -> Vec<gram_diagnostics::Diagnostic>;
    pub fn lint_file(path: &std::path::Path, options: &LintOptions)
        -> anyhow::Result<Vec<gram_diagnostics::Diagnostic>>;
}
```

Note: `gram-data` re-exports `gram_diagnostics::Diagnostic`; `relate` uses that type
directly without conversion.

### relate's Internal Diagnostic Type

`relate` wraps `gram_diagnostics::Diagnostic` with the source file and language,
rather than duplicating the type:

```rust
// src/lint.rs (internal to relate)

pub enum Lang { Cypher, Gram }

pub struct Diagnostic {
    pub lang: Lang,
    pub source_file: Option<std::path::PathBuf>,
    pub inner: gram_diagnostics::Diagnostic,
}
```

Both `cypher-data` and `gram-data` re-export `gram_diagnostics::Diagnostic`, so no
type conversion is needed — `relate` uses the value directly.

### Output

**Human-readable (default):** Formatted diagnostics grouped by file, with severity
colour-coding, rendered via `ariadne` for consistency with the upstream tools.

**JSON (`--json`):** A JSON array of diagnostic objects, one per finding, matching
the schema used by both `cypher-data` and `gram-data`:

```json
[
  {
    "severity": "warning",
    "rule": "structural/unbounded_relationship",
    "message": "Relationship has no upper bound on path length",
    "file": "queries/find-path.cypher",
    "range": {
      "start": { "line": 3, "column": 12 },
      "end":   { "line": 3, "column": 20 }
    }
  }
]
```

### Exit Codes

| Code | Meaning |
|------|---------|
| 0 | No errors (no warnings if `--strict`) |
| 1 | One or more errors found |
| 2 | relate itself failed (I/O error, unsupported file type, etc.) |

This matches the exit code convention of both upstream CLIs.

### Dependencies on Upstream Lib Targets

Both lib targets are now published (verified 2026-05-10):

- `cypher-data` 0.2.3 — lib target available; `check` → `lint` rename complete.
  Resolves: [gram-data/tree-sitter-cypher#5](https://github.com/gram-data/tree-sitter-cypher/issues/5)
- `gram-data` 0.3.9 — lib target available; `check` → `lint` rename complete.
  Resolves: [gram-data/tree-sitter-gram#10](https://github.com/gram-data/tree-sitter-gram/issues/10)

`Cargo.toml` target (partial — see blocker below):

```toml
cypher-data = "0.2.3"
gram-data = "0.3.9"          # blocked — see gram-data/tree-sitter-gram#13
gram-diagnostics = "0.3.9"  # blocked until gram-data unblocked
ariadne = "0.6"
```

**Blocker**: `gram-data 0.3.9` requires `tree-sitter ^0.25`; `cypher-data 0.2.3` requires
`tree-sitter ^0.26.5`. Both link the same native library and cannot coexist. Tracked in
[gram-data/tree-sitter-gram#13](https://github.com/gram-data/tree-sitter-gram/issues/13).
Until resolved, `relate` uses a local mirror of the diagnostic types and implements Cypher
linting only. Gram linting (`.gram` files, `gram` fences in Markdown) is stubbed.

## Amendment: Markdown / Documentation File Support

`relate lint` should support linting code-fenced Cypher and Gram snippets embedded
in Markdown (and AsciiDoc) documentation files.

### Updated Dispatch Table

| Extension | Cypher fences | Gram fences |
|-----------|--------------|-------------|
| `.cypher` | cypher-data lint | — |
| `.gram`   | — | gram-data lint |
| `.md`, `.adoc` | cypher-data lint (relate extracts fences) | gram-data lint (relate extracts fences) |

Directory walks include `.md` and `.adoc` files alongside `.cypher` and `.gram`.

### Fence Language Tags

`relate lint` recognises the following fenced code block language identifiers:

| Tag | Engine |
|-----|--------|
| `cypher`, `openCypher` (case-insensitive) | cypher-data lint |
| `gram` | gram-data lint |

Fences with any other tag are silently skipped.

### Line Number Offset

When linting a snippet extracted from a Markdown fence, diagnostics from the
upstream lib are relative to the snippet (line 1 = first line after the opening
fence). `relate` must add the fence's 0-based start line to every diagnostic's
`range.start.line` and `range.end.line` before reporting, so that output always
references the actual line in the Markdown file.

```
file.md:23:5: warning [structural/unbounded_relationship]
  Relationship has no upper bound on path length
```

`relate` owns this offset calculation; the upstream libs are unaware of the
enclosing document.

### Fence Extraction (relate-owned)

`relate` extracts code-fenced snippets from Markdown and AsciiDoc using a simple
regex scan — the same approach used internally by `cypher-data` and `gram-data` for
their own Markdown support. No cypherdoc or gramdoc library dependency is needed.

**Markdown:** match fenced code blocks with opening ` ``` ` + language tag:

```
/^```[ \t]*(cypher|openCypher|gram)[ \t]*\n([\s\S]*?)^```[ \t]*$/mi
```

**AsciiDoc:** match `[source,<lang>]` followed by a `----` delimited block.

For each match, `relate` records:
- the extracted source text
- the 0-based line number of the opening fence (for offset calculation)
- the language tag (to select the engine)

## Resolved Questions

- **Aligned `Diagnostic` types**: `cypher-data` now depends on `gram-diagnostics`,
  which defines the shared `Diagnostic`, `Severity`, `Range`, and `Position` types.
  Both crates re-export from `gram-diagnostics`; `relate` will add `gram-diagnostics`
  as a direct dependency and use those types for its internal representation.

- **ariadne dependency**: Add `ariadne` to `relate`'s dependencies for human-readable
  diagnostic output, consistent with the upstream tools.

- **Parallel linting**: Out of scope for v1.

- **AsciiDoc**: `relate` is responsible for identifying code fences in both Markdown
  and AsciiDoc. The fence extraction regex described above covers both formats; no
  upstream dependency is needed.
