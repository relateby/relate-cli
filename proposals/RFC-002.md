# RFC-002: relate lint ‚Äî Unified Linting via cypher-data and gram-data

## Summary

`relate lint` is a unified linting command for `.cypher` and `.gram` files.
Rather than implementing lint rules directly, `relate` wraps the linting engines
developed in `cypher-data` and `gram-data`. This RFC specifies the `relate lint`
interface, the dispatch logic by file type, and the API surface each upstream lib
must expose so that `relate` can call them programmatically.

## Motivation

`cypher-data` and `gram-data` are the canonical implementations of Cypher and Gram
linting respectively. Their rule sets (`.scm` query files), diagnostics, and
extensibility mechanisms will evolve independently of `relate`. `relate` should not
duplicate or fork this logic ‚Äî it should be a consumer.

`relate lint` adds value at the orchestration layer:
- A single entry point for both file types in a mixed project
- Integration with the broader `relate` workflow (e.g., lint before write)
- Consistent output format regardless of which engine ran
- A foundation for CI integration across Cypher and Gram files

## Design

### CLI Interface

The `relate lint` interface (already established in `src/cli.rs`) is:

```
relate lint [OPTIONS] [FILES]...

Arguments:
  [FILES]...   Files or directories to lint (.cypher or .gram).
               Reads from stdin if omitted.

Options:
  -e, --expr <EXPR>   Lint an inline expression instead of a file
      --lang <LANG>   Language for --expr or stdin [default: cypher]
                      [possible values: cypher, gram]
      --json          Output diagnostics as JSON
      --strict        Treat warnings as errors
  -h, --help          Print help
```

The `--lang` flag is new relative to RFC-001's stub. It disambiguates the input
language when linting an inline expression or stdin, where a file extension is not
available. Default is `cypher` since inline Cypher snippets are the common case.

### Dispatch Logic

`relate lint` dispatches by file extension:

| Extension | Engine | Upstream command |
|-----------|--------|-----------------|
| `.cypher` | `cypher-data` lint engine | `cypher lint` |
| `.gram`   | `gram-data` lint engine | `gram lint` (currently `gram check` ‚Äî rename requested in [tree-sitter-gram#10](https://github.com/gram-data/tree-sitter-gram/issues/10)) |
| other     | Error: unsupported file type | ‚Äî |

When a directory is given, `relate lint` walks it recursively and dispatches each
`.cypher` and `.gram` file to the appropriate engine. Other files are silently
skipped (not an error).

For `--expr` and stdin, the `--lang` flag selects the engine.

### Required API: cypher-data lib

`relate` needs the following public surface from a `[lib]` target in `cypher-data`:

```rust
// Proposed public API surface for cypher-data lib

pub mod lint {
    /// Options controlling lint behaviour.
    pub struct LintOptions {
        /// Treat warnings as errors (mirrors --strict flag).
        pub strict: bool,
    }

    /// A single diagnostic produced by the lint engine.
    pub struct Diagnostic {
        pub severity: Severity,
        pub rule: String,
        pub message: String,
        pub range: Range,
    }

    pub enum Severity { Error, Warning, Information, Hint }

    pub struct Range {
        pub start: Position,
        pub end: Position,
    }

    pub struct Position {
        /// 0-based line number.
        pub line: usize,
        /// 0-based UTF-8 byte offset within the line.
        pub column: usize,
    }

    /// Lint a Cypher source string. Returns all diagnostics found.
    pub fn lint_source(source: &str, options: &LintOptions) -> Vec<Diagnostic>;

    /// Lint a file at `path`. Returns all diagnostics found.
    pub fn lint_file(path: &std::path::Path, options: &LintOptions)
        -> anyhow::Result<Vec<Diagnostic>>;
}
```

Notes:
- The `.scm` rule files embedded in `cypher-data` must be included in the published
  crate (either via `[include]` in `Cargo.toml` or compiled-in via `include_str!`).
- `relate` does not need to load custom rule directories in the initial version.
  That can be added later.

### Required API: gram-data lib

`relate` needs the following public surface from a `[lib]` target in `gram-data`.
The module is named `lint` to match the rename requested in
[tree-sitter-gram#10](https://github.com/gram-data/tree-sitter-gram/issues/10):

```rust
// Proposed public API surface for gram-data lib

pub mod lint {
    /// Options controlling lint behaviour.
    pub struct LintOptions {
        /// Treat warnings as errors (mirrors --strict flag).
        pub strict: bool,
    }

    /// A single diagnostic produced by the lint engine.
    /// Intended to be structurally identical to cypher-data's Diagnostic
    /// to simplify unification in relate.
    pub struct Diagnostic {
        pub severity: Severity,
        pub rule: String,
        pub message: String,
        pub range: Range,
    }

    pub enum Severity { Error, Warning, Information, Hint }

    pub struct Range {
        pub start: Position,
        pub end: Position,
    }

    pub struct Position {
        pub line: usize,
        pub column: usize,
    }

    /// Lint a Gram source string. Returns all diagnostics found.
    pub fn lint_source(source: &str, options: &LintOptions) -> Vec<Diagnostic>;

    /// Lint a file at `path`. Returns all diagnostics found.
    pub fn lint_file(path: &std::path::Path, options: &LintOptions)
        -> anyhow::Result<Vec<Diagnostic>>;
}
```

### relate's Internal Diagnostic Type

`relate` maintains its own `Diagnostic` type and converts from each lib's type.
This keeps `relate` loosely coupled to the upstream libs' exact type shapes:

```rust
// src/lint.rs (internal to relate)

pub enum Lang { Cypher, Gram }

pub struct Diagnostic {
    pub lang: Lang,
    pub source_file: Option<std::path::PathBuf>,
    pub severity: Severity,
    pub rule: String,
    pub message: String,
    pub range: Range,
}
```

### Output

**Human-readable (default):** Formatted diagnostics grouped by file, with severity
colour-coding. If `ariadne` is available as a dependency, use it for consistency with
the upstream tools. Otherwise, a simple `file:line:col: severity: message` format.

**JSON (`--json`):** A JSON array of diagnostic objects, one per finding, matching
the schema used by both `cypher-data` and `gram-data`:

```json
[
  {
    "severity": "warning",
    "rule": "structural/unbounded_relationship",
    "message": "Relationship has no upper bound on path length",
    "file": "queries/find-path.cypher",
    "range": {
      "start": { "line": 3, "column": 12 },
      "end":   { "line": 3, "column": 20 }
    }
  }
]
```

### Exit Codes

| Code | Meaning |
|------|---------|
| 0 | No errors (no warnings if `--strict`) |
| 1 | One or more errors found |
| 2 | relate itself failed (I/O error, unsupported file type, etc.) |

This matches the exit code convention of both upstream CLIs.

### Dependencies on Upstream Lib Targets

This RFC is blocked on lib targets being published:

- `cypher-data` lib target: [gram-data/tree-sitter-cypher#5](https://github.com/gram-data/tree-sitter-cypher/issues/5)
- `gram-data` lib target + `check` ‚Üí `lint` rename: [gram-data/tree-sitter-gram#10](https://github.com/gram-data/tree-sitter-gram/issues/10)

Once published, `Cargo.toml` will gain:

```toml
cypher-data = "0.3"   # minimum version that includes lib target
gram-data = "0.4"     # minimum version that includes lib target
```

(Version numbers are illustrative; use whatever is published.)

## Amendment: Markdown / Documentation File Support

`relate lint` should support linting code-fenced Cypher and Gram snippets embedded
in Markdown (and AsciiDoc) documentation files.

### Updated Dispatch Table

| Extension | Cypher fences | Gram fences |
|-----------|--------------|-------------|
| `.cypher` | cypher-data lint | ‚Äî |
| `.gram`   | ‚Äî | gram-data lint |
| `.md`, `.adoc` | cypher-data lint (via cypherdoc) | gram-data lint (via gramdoc, TBD) |

Directory walks include `.md` and `.adoc` files alongside `.cypher` and `.gram`.

### Fence Language Tags

`relate lint` recognises the following fenced code block language identifiers:

| Tag | Engine |
|-----|--------|
| `cypher`, `openCypher` (case-insensitive) | cypher-data lint |
| `gram` | gram-data lint |

Fences with any other tag are silently skipped.

### Line Number Offset

When linting a snippet extracted from a Markdown fence, diagnostics from the
upstream lib are relative to the snippet (line 1 = first line after the opening
fence). `relate` must add the fence's 0-based start line to every diagnostic's
`range.start.line` and `range.end.line` before reporting, so that output always
references the actual line in the Markdown file.

```
file.md:23:5: warning [structural/unbounded_relationship]
  Relationship has no upper bound on path length
```

`relate` owns this offset calculation; the upstream libs are unaware of the
enclosing document.

### Additional Required API: cypher-data lib (cypherdoc)

Beyond plain Cypher linting, `cypher-data`'s lib must also expose cypherdoc
extraction so `relate` can feed Markdown content directly without reimplementing
fence parsing:

```rust
pub mod cypherdoc {
    /// A Cypher snippet extracted from a documentation file.
    pub struct Snippet {
        /// The extracted Cypher source text.
        pub source: String,
        /// 0-based line of the opening fence (` ```cypher `) in the parent document.
        pub fence_start_line: usize,
    }

    /// Extract all Cypher snippets from a Markdown or AsciiDoc source string.
    pub fn extract_snippets(doc_source: &str) -> Vec<Snippet>;
}
```

`relate` calls `cypherdoc::extract_snippets`, then calls `lint::lint_source` on each
snippet, then adds `snippet.fence_start_line` to every diagnostic's range. This is
tracked as an addition to [gram-data/tree-sitter-cypher#5](https://github.com/gram-data/tree-sitter-cypher/issues/5).

### Gram-in-Markdown (gramdoc)

A `tree-sitter-gramdoc` grammar (or an extension to `tree-sitter-cypherdoc` to
cover ` ```gram ` fences) is needed for Gram snippet extraction from documentation.
This is expected to follow the same pattern as `tree-sitter-cypherdoc`. Until it
exists, `.md` files with ` ```gram ` fences produce a `warning: gram fences in
Markdown are not yet supported` notice rather than an error.

## Unresolved Questions

- **Aligned `Diagnostic` types**: Should `cypher-data` and `gram-data` share a
  common `gram-diagnostics` crate so `relate` doesn't need conversion code? Or is
  independent-but-structurally-identical good enough? A shared crate would require
  more coordination but simplifies `relate` and any other future consumers.

- **ariadne dependency**: `relate` currently has no `ariadne` dependency. Should it
  add one for consistent human-readable output, or implement a simpler formatter?

- **Parallel linting**: For large directories, should `relate lint` dispatch files
  to a thread pool (via `rayon`)? Out of scope for v1 but worth noting.

- **AsciiDoc**: AsciiDoc uses `[source,cypher]` block delimiters rather than
  fenced code blocks. Does `tree-sitter-cypherdoc` handle AsciiDoc, or only
  Markdown? Scope to be confirmed with upstream.
