# Data Model: Packaging and Distribution

This feature has no persistent application data model. Instead, it defines the structure of **distribution artifacts** and the **release pipeline state** that flows through GitHub Actions.

---

## Release Artifact

A release artifact is the set of files published to a GitHub Release for a given version tag.

**Fields**:
- `version`: semver string matching the `Cargo.toml` version (e.g., `0.2.0`)
- `tag`: git tag (e.g., `v0.2.0` or `v0.2.0-beta.1`)
- `is_prerelease`: derived from tag — true if tag contains a prerelease suffix
- `archives`: one per target platform (see Platform below)
- `checksums_file`: `relate-checksums.txt` — sha256 of every archive
- `dist_manifest`: `dist-manifest.json` — machine-readable release metadata (generated by cargo-dist)
- `install_sh`: `install.sh` — shell installer script
- `install_ps1`: `install.ps1` — PowerShell installer script

---

## Platform

Represents one build target and its artifact.

| Target triple | Archive name | Binary name |
|---|---|---|
| `x86_64-unknown-linux-gnu` | `relate-x86_64-unknown-linux-gnu.tar.gz` | `relate` |
| `aarch64-unknown-linux-gnu` | `relate-aarch64-unknown-linux-gnu.tar.gz` | `relate` |
| `x86_64-apple-darwin` | `relate-x86_64-apple-darwin.tar.gz` | `relate` |
| `aarch64-apple-darwin` | `relate-aarch64-apple-darwin.tar.gz` | `relate` |
| `x86_64-pc-windows-msvc` | `relate-x86_64-pc-windows-msvc.zip` | `relate.exe` |

Each archive contains:
- The binary (`relate` or `relate.exe`)
- `README.md`

## npm Package

**Wrapper package** (`@relateby/cli`):
- `name`: `@relateby/cli`
- `version`: matches release version (e.g., `0.2.0`)
- `bin`: `{ "relate": "bin/relate.js" }` — the Node.js shim
- `optionalDependencies`: map of platform package name → version

**Platform package** (`@relateby/cli-<platform>`):
- `name`: `@relateby/cli-linux-x64` | `@relateby/cli-linux-arm64` | `@relateby/cli-darwin-x64` | `@relateby/cli-darwin-arm64` | `@relateby/cli-win32-x64`
- `version`: matches release version
- `os`: `["linux"]` | `["darwin"]` | `["win32"]`
- `cpu`: `["x64"]` | `["arm64"]`
- `files`: `["bin/relate"]` or `["bin/relate.exe"]`

**npm dist-tags**:
- `latest`: stable releases (e.g., `0.2.0`)
- `next`: prerelease builds (e.g., `0.2.0-beta.1`)

## Homebrew Formula

Managed entirely by cargo-dist; stored in `relateby/homebrew-tap`.

**Logical fields**:
- `name`: `relate`
- `url`: archive URL for the running platform (macOS only)
- `sha256`: checksum from release
- `version`: release version
- `install_path`: `/usr/local/bin/relate` (or Homebrew prefix)

Updated on stable releases only (`vX.Y.Z` without prerelease suffix).

---

## GitHub Secrets (release pipeline state)

| Name | Purpose | Scope |
|---|---|---|
| `HOMEBREW_TAP_TOKEN` | Push formula to `relateby/homebrew-tap` | `release.yml` |
| `CARGO_REGISTRY_TOKEN` | Publish to crates.io | `release.yml` |
| `NPM_TOKEN` | Publish npm packages | `publish-npm.yml` |

These are configuration values, not runtime data. They are stored in the repository's Actions secrets.

---

## Website State

The website (`website/` in `main`, deployed to `gh-pages`) has one piece of dynamic state:

- **Current version**: a version string embedded in `index.html` (e.g., in a `<span id="version">`) updated by `update-website.yml` on each stable release.

The `install.sh` and `install.ps1` files in `gh-pages` are always the latest generated by cargo-dist — updated in sync with the version string.
